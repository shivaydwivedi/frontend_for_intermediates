# Project Standards

Every project must introduce meaningful intermediate frontend concepts and build on earlier work. Do not repeat beginner projects with different styling.

## Required Qualities

Every project must:

1. Introduce a clear set of intermediate concepts.
2. Build on concepts from earlier projects.
3. Include a genuine starter version.
4. Include a complete reference solution.
5. Include detailed but learner-friendly instructions.
6. Be responsive.
7. Be accessible.
8. Handle errors and edge cases.
9. Include meaningful verification or tests.
10. Avoid unnecessary dependencies.
11. Avoid copying an existing commercial product.
12. Avoid fake functionality presented as real.
13. Avoid secrets, exposed API keys, or unsafe authentication patterns.
14. Be independently runnable.
15. Be independently deployable where practical.
16. Clearly distinguish required features from optional challenges.
17. Use original fictional content unless live public API data is required.
18. Include honest limitations and external-service attribution.
19. Prefer maintainable architecture over clever code.
20. Teach browser fundamentals before hiding them behind abstractions.

## Starter And Solution Separation

The starter should guide the learner without solving the project. The solution should demonstrate a complete, tested, accessible implementation.

Do not create identical starter and solution folders. Do not add starter comments that reveal the full answer.

## Documentation

Every project README must meet the standard in [AGENTS.md](../AGENTS.md). It should be educational without becoming a line-by-line answer key.

## Status Truthfulness

Use these statuses consistently:

- Planned.
- In Progress.
- Complete.
- Blocked.

Do not mark a project complete until required features, documentation, accessibility expectations, and verification are complete.
