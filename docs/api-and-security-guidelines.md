# API And Security Guidelines

Frontend projects can teach useful API and security habits, but they cannot provide backend security by themselves. Be honest about these limits in every relevant project.

## API Key Evaluation

Before choosing a public API:

- Check whether an API key is required.
- Check whether the key is public, restricted, or secret.
- Check rate limits and attribution requirements.
- Check whether the API allows client-side browser usage.
- Check whether usage creates cost or account risk.

Never commit a real key. If a project needs configuration, provide `.env.example` with placeholder values.

## Environment Files

- Commit `.env.example` when variables are needed.
- Never commit `.env`.
- Document how learners should create their own local `.env`.
- Do not imply that a secret is safe merely because it is stored in a frontend environment variable.

## Frontend Credential Limits

Anything shipped to the browser can be inspected by users. Frontend code must not contain private credentials. Public client keys may still need domain restrictions, quotas, and careful documentation.

## Safe Requests

- Build URLs with `URL` and `URLSearchParams` when possible.
- Validate URL protocols before rendering external links.
- Check `response.ok`.
- Handle useful status codes such as 400, 401, 403, 404, 429, and 500.
- Show loading and error states.
- Use `AbortController` for obsolete requests.
- Prevent stale responses from overwriting newer state.
- Respect rate limits.
- Use retries and backoff only when they are appropriate and documented.

## Safe Data Rendering

- Treat API, imported, stored, and query-string data as untrusted.
- Prefer text rendering APIs over HTML injection.
- Avoid unsafe `innerHTML`.
- If HTML rendering is unavoidable, explain why and sanitize with a justified approach.
- Validate data at runtime before using it.

## Imports And CSV

- Validate imported file type, shape, required fields, and row-level values.
- Report invalid rows clearly.
- Do not execute imported content.
- Avoid silently accepting malformed data.

## localStorage Privacy

localStorage is convenient, but it is not private secure storage. Do not store sensitive personal data, tokens, or secrets. Explain what is stored and how to clear it when relevant.

## Dependencies

- Review package purpose and maintenance before adding it.
- Avoid packages with unclear provenance for security-sensitive work.
- Review package scripts.
- Keep lockfiles generated by the package manager.
- Never manually invent a lockfile.

## Mock Authentication And Authorization

Mock authentication is useful for frontend learning, but it is not real authentication.

Frontend-only role checks do not provide real authorization. Real authorization must be enforced by a backend. A user can inspect and modify frontend code, storage, and network requests.

Any project with mock auth or role-aware UI must state that it is a demonstration and does not protect real data.
